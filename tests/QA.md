### Детали дефектов

#### БАГ-001: Navigation пустой payload
- **Воспроизведение**: SITL-R-004 — отправить `nav_state` с координатами, запросить `get_state`
- **Ожидаемый результат**: payload содержит `lat, lon, alt_m`
- **Фактический результат**: `"payload": {}`
- **Влияние**: Limiter не может проверить геозоны, Autopilot не знает позицию, Emergency не срабатывает
- **Статус исправления**: закрыт. Регрессионный тест `TestRegression_BAG001_NavigationGetStateNotEmpty` проходит: `navigation` сохраняет координаты из `nav_state` и возвращает их в `get_state`

#### БАГ-002: Cargo пустой payload
- **Воспроизведение**: SITL-R-014 — запросить `get_state` у cargo и проверить поле `state`
- **Ожидаемый результат**: payload содержит `state`
- **Фактический результат**: `"payload": {}`
- **Влияние**: невозможно подтвердить состояние грузового отсека
- **Статус исправления**: закрыт. Регрессионный тест `TestRegression_BAG002_CargoGetStateNotEmpty` проходит: `cargo` возвращает `state` и `last_change_ts`

#### БАГ-003: Motors DDoS-уязвимость
- **Воспроизведение**: SITL-R-010 — отправить 50 невалидных команд, затем ping
- **Ожидаемый результат**: motors отвечает на ping
- **Фактический результат**: timeout 15 секунд
- **Корневая причина**: предположительно, переполнение канала pending requests
- **Влияние**: Нарушение доступности (DoS) — злоумышленник может заблокировать управление моторами
- **Статус исправления**: закрыт. Регрессионный тест `TestRegression_BAG003_MotorsSurvivesInvalidBurst` проходит: после 80 невалидных команд `motors` отвечает на `ping` и не отправляет невалидные команды в SITL

#### БАГ-004: ОрВД — внешний контрактный топик не отвечает
- **Воспроизведение**: ORVD-R-001 — отправить request/response-запрос `register_drone` на `v1.ORVD.ORVD001.main`; ORVD-R-003 — проверить доступность `systems.orvd_system` и `v1.ORVD.ORVD001.main`
- **Ожидаемый результат**: оба заявленных Gateway-топика принимают сообщения формата `action`, `payload`, `sender`, `correlation_id`, `reply_to`; внешний топик возвращает ответы на `register_drone`, `register_mission`, `authorize_mission`, `request_takeoff`, `send_telemetry`
- **Фактический результат**: `systems.orvd_system` отвечает на `get_status`, но `v1.ORVD.ORVD001.main` уходит в timeout. В логах Gateway зафиксировано: `Error subscribing to v1.ORVD.ORVD001.main: NoBrokersAvailable`
- **Дополнительное наблюдение**: README ОрВД одновременно описывает старые топики вида `drone/mission/request`, `drone/takeoff/request`, `ordv/authorization`, общую схему `systems.<имя_системы>` и внешний API `v1.ORVD.ORVD001.main` в коде `topics.py`
- **Корневая причина**: предположительно, Gateway подписывается на внешний топик до готовности Kafka; после восстановления брокера `BaseSystem.start()` подписывает только системный топик `systems.orvd_system`, а внешний топик не переподписывается
- **Влияние**: дрон не может надёжно интегрироваться с ОрВД по внешнему контракту; невозможно проверить авторизацию вылета, запрет взлёта до авторизации и обмен телеметрией через заявленный внешний топик

#### БАГ-005: Motors отправляет команды SITL в несовместимом формате
- **Воспроизведение**: SITL-001/SITL-002 — отправить `SET_TARGET` или `LAND` в `motors` от доверенного `security_monitor`; SITL-015 — отправить `SET_TARGET` через `security_monitor` от `autopilot`
- **Ожидаемый результат**: в топик SITL commands публикуется плоский JSON по схеме SITL: `drone_id`, `vx`, `vy`, `vz`, `mag_heading`
- **Фактический результат**: публикуется вложенное сообщение вида `{"source":"motors","command":{"cmd":"SET_TARGET","target":{...}}}` или `{"source":"motors","command":{"cmd":"LAND"}}`
- **Влияние**: SITL verifier/controller не может принять команду по своему контракту; сценарии управления полётом через SITL не подтверждаются
- **Ответственность**: дефект на стороне системы дрона, потому что именно `motors` формирует сообщение на внешний SITL-топик
- **Статус исправления**: открыт


#### БАГ-006: Security Monitor не журналирует allow/deny решения
- **Воспроизведение**: SEC-008 — запустить `journal`, `motors`, `security_monitor`; выполнить одну разрешённую политикой команду и одну запрещённую команду через `proxy_publish`
- **Ожидаемый результат**: в журнале появляются события решений контроля доступа с `decision=allow` и `decision=deny`
- **Фактический результат**: журнал остаётся пустым для обычных решений доступа; при этом SEC-007 подтверждает, что переход `ISOLATED` логируется
- **Влияние**: невозможно доказать выполнение ЦБ-4: аудит решений контроля доступа неполный
- **Ответственность**: дефект на стороне системы дрона, потому что `security_monitor` принимает решения политик и должен передавать их в `journal`
- **Статус исправления**: открыт


#### Примечание по границам SITL-покрытия
- Переписанные SITL-тесты запускают реальные компоненты дрона на реальном MQTT-брокере с изолированными тестовыми топиками
- Проверки `SITL-005`–`SITL-014` из общего списка относятся к внутренней логике SITL/DronePort (`HOME`, TTL, core movement, position request) и не оформляются как дефекты команды дрона
- Для нашей команды (дрон) критична брокерная граница: формат команд `motors -> SITL` и приём `nav_state` в Navigation


### Внешние замечания к соседним системам

- **GCS-R-005 (НУС):** `task.assign` для несуществующего `mission_id` возвращает успешный сценарий загрузки миссии. Это замечание к НУС и не относится к дефектам команды дрона.
- **DronePort-GCS-020:** DroneManager не поддерживает заявленный сценарий `request_charging` и возвращает `Unknown action: request_charging`. В реализации зарядку можно запустить через `charging_manager.start_charging`, но контракт GCS-020 описывает запрос через DroneManager, поэтому требование не подтверждено.
- **Наблюдение DronePort:** `request_landing` публикует регистрацию дрона и обновление батареи асинхронно; при неблагоприятном порядке обработки в `drone_registry` может появиться неполная запись без `port_id`. В тесте вылета это вынесено в precondition, чтобы GCS-019 проверял именно вылет и освобождение порта.
