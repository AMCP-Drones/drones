# Отчет по ДВБ и кибериммунности (актуализация)

Дата актуализации: 11 мая 2026

---

## 1) [разработчики] Оценка размера ДВБ

Методика: посчитаны непустые строки production Go-кода (`*.go`), без `vendor`, `tests`, `.generated`, `*_test.go`.

В рамках текущего scope ЦБПБ (**message-control + trusted executors**) ДВБ определяется как:
`security_monitor + component + config + journal + navigation + motors + cargo`.

**Итого размер ДВБ: 1734 LOC (~1.73 KLOC)**.

Разбиение по доменам ДВБ:

- Политики и доверие (`security_monitor`, `component`, `config`) — **1050 LOC**
- Журналирование и аудит решений доступа (`journal`) — **188 LOC**
- Доверенные исполнительные домены (`navigation`, `motors`, `cargo`) — **496 LOC**

Справочно (вне текущей ДВБ для данного scope): полный production-код системы — **4422 LOC (~4.42 KLOC)**.

### Что изменилось по структуре кода

Реализовано 5 архитектурных улучшений для кибериммунного контура:

1. Общий secure-proxy клиент (`component/security_proxy.go`)
2. Общий аудит-логгер (`component/audit_logger.go`)
3. Табличная state machine для `autopilot`
4. Унификация control/poll loop (`component/loop.go`)
5. Декомпозиция `security_monitor` на policy/proxy/isolation

Эффект: уменьшены крупные "монолитные" файлы (`autopilot`, `limiter`, `mission_handler`), но добавлены общие модули переиспользования и явные механизмы безопасности. Это повышает проверяемость ДВБ и снижает дублирование.

### Эффект нового паттерна на размер ДВБ

Введен новый паттерн: **«Выделенный механизм поддержания состояния безопасности»** (`security_monitor` как единый держатель состояния + детерминированные переходы + восстановление).

Наблюдаемый эффект по коду:

- Критичный функционально-безопасный контур (`limiter` + `emergency`) снижен до **506 LOC** (было **555 LOC**, ~**-8.8%**).
- Часть логики перемещена в централизованный слой политик/состояний (`security_monitor`), что уменьшает дублирование и упрощает аудит.
- В текущем scope (message-control + trusted executors) safety-governance централизована в `security_monitor`, а исполнительные модули `navigation/motors/cargo` формально включены в ДВБ.

### Сопоставление с паттерном «Выделенный механизм поддержания состояния безопасности» (`cyber_drons`)

При сравнении с соседней системой `../cyber_drons` зафиксированы и перенесены следующие технические отличия:

1. Единое каноническое состояние безопасности в `security_monitor` (mode/reason/source/transition_id/version).
2. Явные переходы между режимами `NORMAL` и `ISOLATED` с отдельными действиями `ISOLATION_START` и `ISOLATION_END`.
3. Идемпотентность переходов (повторная активация/деактивация не создает дублирующих переходов и журналов).
4. Разделение базовых политик (`normalPolicies`) и активных политик (`policies`) с восстановлением после выхода из изоляции.
5. Усиление доверенной идентичности отправителя: проверка на точное соответствие sender, без prefix-match.
6. Добавление watchdog-семантики в `security_monitor` (heartbeat timeout + failsafe isolate) с выключенным по умолчанию режимом.
7. Расширенный аудит переходов режима безопасности в журнале с метаданными перехода (`transition_id`, `state_version`, `reason`, `trigger_source`, `transition_at`).

## 3) [архитектор] Актуализированные ЦБПБ команды БАС

### 3.1 Цели безопасности (ЦБ)

1. Выполняются только аутентичные и авторизованные команды управления полётом и перемещением БАС; при невозможности установить подлинность/полномочия команда отклоняется.
2. Выполняются только аутентичные и авторизованные операции с грузом (открытие/закрытие/выдача) в рамках активной и разрешённой миссии.
3. Выполняются только аутентичные и авторизованные полётные задания; задания, не прошедшие проверку подлинности источника и ограничений ОРВД, не принимаются к исполнению.
4. Все значимые события безопасности и переходы в аварийные/изоляционные режимы фиксируются в защищённом журнале для последующего анализа.
5. Полёт и работа с грузом выполняются с соблюдением ограничений миссии (геозоны, высота, этапы, режимы); при нарушении ограничений инициируется fail-safe сценарий.
6. Обмен критичными данными с внешними контурами (ОРВД/НУС/Дронопорт) обеспечивает целостность и подлинность; конфиденциальность обеспечивается для данных, отмеченных как конфиденциальные в модели угроз.
7. Конструкция грузового отсека и крепления груза обеспечивает удержание груза и предотвращение неконтролируемого выпадения в пределах эксплуатационных воздействий.
8. ПО и конфигурации ДВБ исполнительных контуров имеют проверяемое происхождение и контролируемую целостность, исключая несанкционированные модификации.

### 3.2 Предположения безопасности (ПБ)

1. Внешние системы (ОРВД/НУС/Дронопорт) предоставляют корректные криптографические материалы и поддерживают согласованные механизмы аутентификации.
2. Каналы связи БАС с внешними контурами реализуют согласованный защищённый транспорт (TLS/VPN/эквивалент).
3. Инфраструктура эксплуатации предоставляет средства безопасного хранения, ротации и отзыва секретов/сертификатов.
4. Наземные сервисы передают БАС данные в согласованных форматах и с корректными атрибутами полномочий.

## 4) [архитектор] Состояние шаблонов СКИБ

Подтвержденные паттерны:

- Reference Monitor / PEP — `security_monitor` как центральный посредник
- Mediation-only access — критичные действия идут через `proxy_request` / `proxy_publish`
- Defense in Depth — `security_monitor + limiter + emergency + journal`
- Fail-safe default — deny-by-default при отсутствии разрешающей политики
- Dedicated Safety-State Mechanism — централизованная модель `NORMAL/ISOLATED` с управляемыми переходами в `security_monitor`

Дополнительно усилено в этой итерации:

- Централизованы security-операции (proxy + audit), снижена дублируемая обвязка
- Упорядочены переходы состояний в `autopilot` (табличный формат)
- Повышена декомпозируемость и тестопригодность `security_monitor`
- Добавлен явный recovery-path (`ISOLATION_END`) и metadata-аудит переходов режима безопасности

Ограничения (остаются):

- Нет криптографической аутентификации/целостности сообщений (по условию задачи не внедрялось)
- Нет покрытия ЦБ-8 (доверенная поставка ПО/SBOM-процесс)
- Интеграции ОрВД/GCS/DronePort остаются отдельным контуром

## 6) Вывод

- По default-контуру кибериммунных механизмов достигнут заметный прогресс:
  - архитектура стала более модульной и проверяемой,
  - введен dedicated safety-state паттерн с централизованными переходами и recovery,
  - функционально-безопасный контур (`limiter`+`emergency`) уменьшен на ~**8.8%**,
  - покрытие ДВБ без транспортных адаптеров сейчас **74.4%**,
  - закрыта значимая часть негативных сценариев.
- Для "зрелой" оценки курса еще нужны: добор покрытия до целевого уровня, интеграционный прогон SITL после рефакторинга, фиксация старых SITL-дефектов и отдельный quality gate по ДВБ + integration.
